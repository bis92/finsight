# Architecture Decision Records

## 철학
MVP 속도 최우선. 외부 의존성은 "한 번에 여러 문제를 푸는" 것으로 최소화한다(Supabase = Auth+DB+Storage, Polar = 결제+세금). 작동하는 최소 구현을 선택하되, **비용이 사용자 수에 비례하는 LLM 호출은 처음부터 통제 설계**를 넣는다.

---

### ADR-001: Next.js 15 App Router 풀스택
**결정**: 프론트·백엔드를 Next.js 한 코드베이스로. Route Handler로 서버 로직 처리.
**이유**: 랜딩·대시보드·API·웹훅을 한곳에서. Vercel 배포가 무설정에 가깝고 MVP 속도 최적.
**트레이드오프**: 무거운 백엔드/장기 실행 작업엔 부적합. CSV 분석은 요청-응답 범위 안이라 문제없음.

### ADR-002: Supabase (Auth + Postgres + Storage)
**결정**: 인증·DB·파일저장을 Supabase 하나로.
**이유**: 세 인프라를 각각 붙일 시간을 절약. RLS로 유저별 데이터 격리를 DB 레벨에서 강제.
**트레이드오프**: 벤더 종속. RLS 정책을 정확히 짜야 함(틀리면 데이터 노출).

### ADR-003: LLM = Claude API (추상화 얇게)
**결정**: 기본 LLM은 Claude(`@anthropic-ai/sdk`). `services/llm`에 얇은 인터페이스를 두되 MVP는 Claude 단일 구현.
**이유**: 긴 CSV 컨텍스트 처리와 구조화 JSON 출력이 안정적. 프로바이더 교체 여지는 남기되 과설계는 피함.
**트레이드오프**: 멀티 프로바이더 런타임 전환은 MVP 범위 밖.

### ADR-004: 결제 = Polar (Merchant of Record)
**결정**: 구독 결제는 Polar. Checkout 링크 + 웹훅으로 `profiles.plan` 갱신.
**이유**: MoR라 VAT/세금 처리를 대행. Next.js 연동이 단순하고 인디/SaaS 구독에 적합.
**트레이드오프**: 국내 PG 대비 결제수단 커버리지 제약 가능. MVP 검증 단계엔 수용.

### ADR-005: LLM 비용 통제 — 컬럼매핑만 LLM, 분류는 규칙+배치
**결정**: CSV 전체를 LLM에 넣지 않는다. 컬럼 스키마 인식은 **샘플 행(≤20)**만 LLM에 전달하고, 전체 거래의 카테고리 분류는 규칙 기반(+필요시 소량 배치 LLM)으로.
**이유**: 업로드마다 수백~수천 행을 LLM에 통째로 넣으면 비용·지연이 유저 수에 비례해 폭증. 매핑 스키마는 파일당 1회면 충분.
**트레이드오프**: 규칙 분류 초기 정확도는 LLM 전량 분류보다 낮을 수 있음. 카테고리 보정은 후속 개선으로.

### ADR-006: Free/Pro 경계 = 기능 깊이 (횟수 아님)
**결정**: 무료/유료를 사용 횟수가 아니라 분석 깊이로 가른다. Free=집계 조회, Pro=LLM 해석·정기구독 탐지.
**이유**: 명세서는 월 1회 단위라 횟수 제한이 페이월로 작동하지 않음. 무거운 LLM 기능을 Pro 뒤에 두면 매출이 비용을 커버.
**트레이드오프**: Free만으로도 충분하다고 느낄 위험 → Pro 앵커(인사이트·구독탐지)를 "한 번 쓰면 계속 쓰는" 기능으로 배치해 상쇄.
