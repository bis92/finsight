# Architecture Decision Records

> 계획의 단일 원천은 이 문서(및 `PRD.md`·`ARCHITECTURE.md`)다.

## 철학

개인 금융 PII를 다루는 **한국 B2C SaaS**. 보안·데이터 무결성을 최우선으로 하되, mock-first 시임으로 UI를 먼저 완성하고 실연동은 repository 교체로 처리해 UI 코드를 불변으로 유지한다. 핵심 가치(AI 지출 진단)를 먼저 벽까지 관통시키고, LLM은 얇게 추상화하며 비용을 구조적으로 통제한다. TDD로 신뢰 가능한 코어부터 쌓는다.

---

### ADR-001: Next.js 15 (App Router) 풀스택 + Vercel 배포
**결정**: Next.js 15 App Router로 페이지 + Route Handler를 한 코드베이스에서, Vercel에 `vercel` CLI로 자동 배포.
**이유**: 프론트/백엔드 단일 스택으로 MVP 속도 확보. 외부 API 호출을 서버(Route Handler)에 가둬 시크릿·비용을 통제. Vercel은 Next.js 1급 지원.
**트레이드오프**: 서버리스 콜드스타트·벤더 락인. 무거운 백그라운드 작업엔 부적합(현재 범위엔 문제 없음).

### ADR-002: Supabase (Auth + Postgres + Storage, RLS) · 카카오 + 구글 OAuth
**결정**: 인증·DB·스토리지를 Supabase로 통합. 모든 테이블 RLS `user_id = auth.uid()`. 로그인은 **카카오 + 구글 OAuth**(이메일/비밀번호 없음).
**이유**: 세 가지를 한 곳에서 + RLS로 크로스테넌트 격리를 DB 레벨에서 강제. 한국 소비자 대상이라 카카오로 가입 장벽을 최소화하고 전환율을 높인다. 비밀번호 관리 불필요.
**트레이드오프**: 벤더 종속. 카카오는 Supabase 기본 provider가 아니라 커스텀 OIDC 설정이 필요(초기 셋업 비용). service-role 키 오남용 시 RLS 우회 위험 → **service-role은 Polar 웹훅 plan 갱신에만** 사용으로 제한.

### ADR-003: Claude API (`claude-opus-4-8`), 얇은 추상화
**결정**: LLM은 Claude API(`@anthropic-ai/sdk`), 모델 `claude-opus-4-8`(Opus 4.8) 고정. `LlmService` 인터페이스로 얇게 추상화. 어댑티브 씽킹(`thinking:{type:"adaptive"}`), effort는 작업별 차등, 구조화 출력은 `output_config.format`.
**이유**: 컬럼 매핑·지출 진단 리포트에 고품질 추론이 필요. 얇은 추상화로 mock/live 교체와 프롬프트 격리.
**트레이드오프**: 단가 $5/$25 per 1M(Sonnet의 ~1.67배) → 비용 통제책(ADR-004)이 필수 전제. `budget_tokens`·`temperature`·마지막 assistant 프리필은 400(사용 금지).

### ADR-004: 컬럼 매핑은 AI, 거래 분류는 규칙 기반 + 배치 (LLM 비용 통제)
**결정**: 카드사별 포맷을 하드코딩하지 않고 헤더+샘플 ≤20행을 Claude에 보내 컬럼(날짜/가맹점/금액/카테고리)을 추론시키고, 사용자 확인·수정 UI를 둔다. **CSV 전체 행은 LLM에 넣지 않는다** — 실제 거래 분류는 규칙 기반 + (필요시 소량) 배치. LLM 호출은 기본 경로 3회 이하로 상한.
**이유**: 국내 카드사마다 컬럼명·인코딩·날짜형식이 제각각이라 하드코딩 템플릿은 미지원 카드사에서 즉시 실패하고, AI 매핑은 범위가 넓다. 수백~수천 행을 매 업로드마다 LLM에 넣으면 비용·지연이 폭증한다.
**트레이드오프**: 매핑 오류 가능성·토큰 비용 → 사용자 확인 UI + 샘플 행만 전송으로 억제. 규칙 기반 분류의 정확도 한계 → 재분류 Modal(category enum)로 사용자 보정.

### ADR-005: 집계는 코드, 판단은 AI (구조화 JSON 출력)
**결정**: 카테고리 합계·비율 등 숫자는 `lib/`에서 결정론적으로 계산. Opus 4.8은 정해진 JSON 스키마로 진단문·절감제안 배열만 반환.
**이유**: LLM에 숫자 계산을 맡기면 환각·불일치가 발생한다. 차트는 코드 집계로 정확히 그리고, 텍스트만 AI가 생성하면 신뢰도·테스트 용이성이 높아진다.
**트레이드오프**: 스키마 설계·검증 코드가 필요. 자유 서술의 유연함은 일부 포기.

### ADR-006: 과금 = 기능 깊이로 가름 (+ Free 월 5회 안전장치)
**결정**: Free(사실 조회)/Pro(해석·코칭)를 기능 깊이로 구분. Free는 대시보드 기능 전부 + 업로드 **월 5회**(비용 안전장치), Pro는 업로드·분석 무제한. Pro 앵커는 **지출 진단 리포트 + 정기구독 후보 탐지**.
**이유**: 명세서는 월 1회라 순수 횟수 게이팅은 무의미. Free=집계(LLM 가벼움), Pro=추론(LLM 무거움)이라 매출이 LLM 비용을 커버. Free 업로드 월 5회 제한은 수익 게이팅이 아니라 매핑 LLM 비용 폭주 방지용.
**트레이드오프**: Free만으로도 상당한 가치를 제공 → 전환 설득이 페이월 UX에 달림.

### ADR-007: 결제 = Polar (MoR), 실결제는 phase 1 · 통화 리스크 명시
**결정**: 구독 결제는 Polar(Merchant of Record). 권한의 유일한 진실 원천은 웹훅으로 갱신된 `profiles.plan`. **phase 0은 게이팅 구조만(mock), 실결제 연동은 phase 1.**
**이유**: MoR가 부가세·인보이스를 대행. 웹훅 단일 경로로 plan을 갱신해 셀프 업그레이드를 차단. mock-first(ADR-008)와 정합.
**트레이드오프**: ⚠️ Polar는 국내카드·원화 지원이 제한적이라 한국 타깃과 긴장이 있고 실제 결제 가능층이 좁아질 수 있음. phase 1 실연동 시 국내 PG(토스페이먼츠/포트원) 전환을 재검토. 웹훅 순서 뒤섞임 대응은 MVP 밖(단순 멱등만).

### ADR-008: mock-first 시임 (`DATA_SOURCE` 스위치)
**결정**: mock 데이터는 `/api/*` 뒤 repository(`services/`)에만. `DATA_SOURCE=mock|live` 서버 env로 전환. UI/queries는 phase 0에서도 실제 `/api`를 호출.
**이유**: UI를 먼저 완성하고 실연동 시 repository 구현만 교체해 UI 코드를 불변으로 유지. mock 상태로 먼저 배포해 핵심 루프를 조기 검증.
**트레이드오프**: mock/real 반환 타입을 완전히 동일하게 유지하는 규율이 필요. 스텁 인증이 공개 mock 배포에 노출되므로 `DATA_SOURCE` 기준 가드로 live에서 fail-closed.

### ADR-009: 원본 거래 저장 + RLS·at-rest 암호화·완전삭제 (컬럼 암호화 제외)
**결정**: 집계 결과뿐 아니라 원본 거래내역을 Supabase에 저장한다. 모든 테이블 RLS(`user_id = auth.uid()`), Supabase at-rest 암호화 + 비공개 Storage, 회원 탈퇴 시 완전 삭제(삭제권)를 필수 요구사항으로 둔다. **앱단 컬럼 암호화는 하지 않는다.**
**이유**: 재방문 시 과거 리포트·거래를 보여줘 구독 가치를 유지하고, 정기구독 탐지·누적 분석의 기반이 된다. `amount`/`merchant`를 앱단 암호화하면 대시보드 집계를 DB에서 `SUM`/`GROUP BY`로 수행할 수 없으므로, 기밀성은 RLS + at-rest + 접근통제로 확보하고 집계 성능·단순성을 지킨다.
**트레이드오프**: 민감 금융정보 유출 리스크와 보안 책임이 커진다 → RLS·at-rest·삭제권으로 방어. 앱단 컬럼 암호화만큼의 저장 기밀성은 포기(위협모델상 RLS+at-rest로 충분하다고 판단).
